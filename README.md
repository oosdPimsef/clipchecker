# Clipchecker

Локальный MVP для предобработки рекламных видеоматериалов и подготовки материалов для юридического анализа.

## Текущий запуск

Основной интерфейс запускается из папки `app`:

```powershell
python main.py
```

После запуска форма доступна по адресу:

```text
http://localhost:8080/
```

Для анализа нужно указать три папки:

- папка с превью-роликом;
- папка с документами;
- папка с эфирной версией ролика.

В форме есть раскрываемый блок `Материалы для тестирования`. В нём можно сохранить три тестовых пути; они хранятся в браузере через `localStorage` и не участвуют в запуске анализа, пока пользователь не нажмёт `Копировать ссылки`. После копирования значения переносятся в основные поля формы.

Также в форме выбирается категория рекламодателя. Список категорий загружается из файла:

```text
C:\Users\PTambulatov\Desktop\PT\2.Расчеты\50. ХАКАТОН\14. карта согласования рм\1. Карта согласования РМ.xlsx
```

На экране результатов после онлайн-лога выводится карта согласования РМ для выбранной категории. Позиции берутся строго сверху вниз по `id` в столбце D, начиная со строки 3; строки с пустым `id` игнорируются. Блоки формируются в том порядке, в котором они впервые встречаются при таком проходе по файлу.

Каждая позиция карты согласования получает `id` из одноимённого столбца Excel. Дальше Python-оценщики нужно привязывать именно к этому `id`, а не к тексту проверки или порядковому номеру.

Реализованные Python-оценщики:

- `id=1`: проверяет, что длительность ролика кратна 5 секундам. Длительность берётся из предобработанных кадров, которые извлекаются с частотой 1 кадр в секунду.
- `id=2`: ищет в `Documents_Texts.txt` БЗ / Бланк-заявку, извлекает длительность ролика из текста БЗ и сравнивает её с длительностью ролика из `id=1`. Если БЗ нет или длительности не совпадают, позиция подсвечивается красным.
- `id=3`: проверяет по `OCR_Log.json`, что текст юридических набивок в нижней части кадра находится в зелёной рамке.
- `id=4`: проверяет, что логотипы/брендовые элементы рекламодателя находятся в зелёной рамке. Используется OCR для короткого повторяющегося брендового текста и локальная YOLOWorld CV-модель для визуальных логотипов без OCR-текста.
- `id=5`: проверяет, что все надписи и существенные визуальные элементы на экране находятся в зелёной рамке: набивки, цены, возрастные маркировки, OCR-тексты, логотипы, упаковки, плашки, баннеры и другие CV-объекты.
- `id=6`: проверяет явные орфографические/OCR-ошибки в словах вне юридической набивки.
- `id=7`: проверяет явные орфографические/OCR-ошибки в словах юридической набивки в нижней части кадра.
- `id=8`: ищет иностранные слова на кадрах по `OCR_Log.json`. Если слова найдены, пункт получает оранжевый статус `warning`, а найденные слова выводятся в интерфейсе жирным красным шрифтом с предупреждением: `Иностранные слова должны быть переведены или предоставлены документы на товарный знак!`
- `id=9`: проверяет по исходным извлечённым кадрам, что по бокам нет устойчивых черных вертикальных полос.
- `id=10`: ищет цену в рублях в OCR-тексте кадров и набивок. Найденные цены выводятся крупными жирными красными цифрами. Отсутствие цены не считается ошибкой; красный статус ставится только если цена в основном кадре не совпадает с ценой в юридической набивке.
- `id=11`: ищет цену товара в `Documents_Texts.txt`; сначала проверяет документы с названием вроде `Письмо о ценах`, `справка о стоимости`, `ценовое предложение`, затем остальные документы. Поддерживает свободные формулировки `ценовое предложение`, `установлено в размере`, `стоимость товара`, `акционная цена`, `розничная цена`, а также суммы с прописью в скобках перед `рублей`. Найденная цена выводится крупным жирным красным шрифтом. Если цена есть только в документах, а в видеоряде её нет, статус зелёный; если цена в документах и видеоряде не совпадает, статус красный.
- `id=12`: ищет признаки ювелирных изделий на кадрах по OCR-тексту (`OCR_Log.json`, резервно `All_Frames_Text.txt`): `кольцо`, `серьги`, `браслет`, `цепочка`, `подвеска`, `золото`, `бриллиант`, `jewelry`, `ring` и т.п. Если признаки найдены, статус зелёный и выводится сообщение о необходимости документов по спецучёту. Если признаки не найдены при наличии предобработанных кадров, статус оранжевый и выводится рекомендация вручную проверить наличие ювелирных изделий и документы ГИИС ДМДК. Полноценное CV-распознавание визуального объекта будет отдельным усилением этого пункта.
- `id=13`: использует ту же детекцию ювелирных изделий, что `id=12`. Если признаки ювелирных изделий найдены, статус оранжевый и выводится сообщение `Необходимо предосатвить бирки на ювелирную продукцию в ролике`. Если признаки не найдены, статус зелёный и выводится сообщение `Ювелирных изделий в видеоряд не найдено`.
- `id=14`: берёт изображения лиц из `Clipchecker_materials/faces`, то есть из исходников для `Faces_Thumbnails.pdf`, и выполняет отдельный сетевой поиск имён без Telegram и без `Face_Recognition_Results.txt`. Результат кэшируется в `Actor_Web_Search_Results.json`. Если лиц нет, статус зелёный и выводится `Актеров в ролике нет`. Если лицо есть, но имя не определено, статус зелёный и выводится `Актер не распознан`. Если имя определено, статус зелёный, а имя выводится жирным красным шрифтом. Для подключения реального внешнего сервиса поиска по фото задаётся `MEDIA_PERSON_WEB_SEARCH_ENDPOINT`; без него лица считаются нераспознанными.
- `id=16`: проверяет OCR-текст всех кадров по словарю проверяемых claims: `№1`, `лучший`, `самый`, `эффективный`, `безопасный`, `новинка`, `официальный`, `рекомендовано`, `результаты исследований`, `скидка до`, `цена от`, `лидер`, `уникальный`, `доказано`, `подтверждено`, `хит продаж`, `выбор покупателей` и т.д.; словарь содержит больше 50 позиций. Если совпадений нет, статус зелёный и выводится `Проверяемые claims не обнаружены`. Если совпадения есть, статус красный, найденные claims выводятся жирным красным шрифтом с номерами кадров.
- `id=235`: использует найденные `id=16` claims и проверяет, есть ли по ним объясняющая юридическая набивка внизу кадра или подтверждающие фразы в `Documents_Texts.txt` (`по данным`, `согласно`, `исследование`, `сертификат`, `заключение`, `условия акции`, `подробности`, `цена от`, `скидка до` и т.п.). Если claim нет, статус зелёный. Если claim есть и есть набивка или документ, статус оранжевый, в интерфейс выводятся сам claim, текст набивки и/или фраза из документов. Если claim есть, а набивки и документа нет, статус красный.
- `id=246`: ищет в тексте юридической набивки фразу `Дистанционные продажи` в разных склонениях (`дистанционных продаж`, `дистанционным продажам` и т.п.). Если фраза найдена, статус зелёный и выводится текст набивки с секундами. Если фраза не найдена, статус оранжевый.
- `id=247`: ищет в тексте юридической набивки сайт, страницу в интернете или номер телефона. Поддерживает URL/домены (`example.ru`, `www.example.ru/page`, `https://...`), страницы мессенджеров/соцсетей и российские номера `+7...` / `8...`. Если найден хотя бы один контакт, статус зелёный, найденные адреса и телефоны выводятся жирным красным шрифтом. Если сайт и телефон не найдены, статус оранжевый.
- `id=249`: проверяет OCR-текст кадров и, при наличии локальной YOLO-модели, сами изображения кадров на признаки алкоголя: алкоголь, спиртное, водка, пиво, вино, шампанское, виски, коньяк, коктейль, бар, бокал, бутылка, банка пива и т.п. Если признаки найдены, статус красный, найденные слова или CV-объекты выводятся жирным красным шрифтом с номерами кадров.
- `id=252`: ищет на всех OCR-кадрах слова, не принадлежащие русскому языку: латиницу, смешение латиницы и кириллицы, а также частые англицизмы/неологизмы на кириллице (`кэшбэк`, `онлайн`, `маркетплейс`, `шопинг`, `фэшн`, `лайфстайл`, `хайп` и т.п.). Если слов нет, статус зелёный. Если слова найдены, статус красный, найденные слова выводятся жирным красным шрифтом с секундами.
- `id=258`: связан с `id=252`. Если англицизмы/неологизмы не найдены, статус зелёный. Если слова найдены и на кадрах есть перевод со звёздочкой, статус оранжевый, выводятся найденные слова и текст перевода. Если слова найдены, а перевода со звёздочкой нет, статус красный.
- `id=228`: проверяет отсутствие стоп-кадров. Программа последовательно сравнивает соседние предобработанные кадры из `frames_pdf_original` / `frames`; если сходство пары кадров не ниже 90%, пункт получает красный статус и выводит номера повторяющихся кадров. Если повторов нет, статус зелёный.
- `id=229`: проверяет отсутствие пустых однотонных кадров любого цвета, включая чёрные, белые и серые. Если такие кадры найдены, пункт получает красный статус и выводит номера кадров; если пустых кадров нет, статус зелёный.
- `id=251`: проверяет наличие государственных, муниципальных, религиозных и близких к ним символов на кадрах. Словарь содержит больше 15 определений: флаг, государственный флаг, герб, герб города/региона, двуглавый орёл, гимн, штандарт, эмблемы госорганов, символы власти, георгиевская лента, военная символика, крест, полумесяц, звезда Давида, икона и т.д. Дополнительно используется визуальная эвристика для триколора и YOLOWorld CV-поиск флагов, гербов, эмблем и религиозных символов. Если признаков нет, статус зелёный и выводится `Государственные символы не обнаружены`. Если признаки есть, статус красный, найденные элементы выводятся жирным красным шрифтом с номерами кадров.
- `id=262`: связан с `id=8`. Если иностранных слов нет, статус зелёный. Если иностранные слова найдены, программа ищет в OCR строку со звёздочкой как перевод и проверяет `Documents_Texts.txt` на документы по товарному знаку. При наличии перевода или документа статус оранжевый, в интерфейс выводится текст перевода или название бренда из документа. Если иностранные слова есть, а перевод и права на товарный знак не найдены, статус красный.
- `id=269`: проверяет OCR-текст кадров и CV-объекты на признаки курения и табака: сигареты, табак, никотин, папиросы, сигара, вейп, электронная сигарета, кальян, дым, затяжка, пепельница, зажигалка и т.п. Если признаки найдены, статус красный.
- `id=270`: проверяет OCR-текст кадров и CV-объекты на признаки запрещённых веществ: наркотики, каннабис, марихуана, гашиш, героин, кокаин, амфетамин, экстази, ЛСД, спайс, доза, дилер, шприц, порошок и т.п. Если признаки найдены, статус красный.
- `id=271`: проверяет OCR-текст кадров и CV-объекты на признаки оружия или насилия: оружие, пистолет, автомат, ружьё, патроны, пуля, снаряд, граната, бомба, нож, кинжал, стрельба, выстрел, кровь, драка, нападение и т.п. Если признаки найдены, статус красный.

Для CV-проверки `id=4/5/249/251/269/270/271` используется локальная YOLOWorld-модель из переменной окружения `CLIPCHECKER_YOLO_MODEL` или `RESTRICTED_CONTENT_YOLO_MODEL`. На локальном MVP файл модели лежит в `app\models\yolov8s-world.pt`, а `CLIPCHECKER_YOLO_MODEL` задан на этот путь. CLIP-кэш для open-vocabulary режима хранится локально в `app\models\cv_cache`; программа больше не пишет в заблокированный пользовательский `.cache`. Порог CV по умолчанию `0.55`, его можно изменить через `RESTRICTED_CONTENT_CV_CONFIDENCE`. `id=9`, `id=228` и `id=229` остаются пиксельными проверками без YOLO, потому что они анализируют геометрию полос, повторы и однотонность кадров. `id=10` остаётся OCR-проверкой цены, так как цену нужно извлекать как текст.

Для `id=6` и `id=7` сокращения и написания через дефис не считаются автоматической ошибкой, но выводятся в рекомендации для ручной проверки.

## Извлечение текста документов

Предобработка документов сохраняет полный собранный текст в `Documents_Texts.txt`. Для `.docx` учитываются не только обычные абзацы, но и текст внутри таблиц, потому что БЗ часто приходит именно в табличной форме.

Бланк-заявка для проверки `id=2` определяется по названию документа или содержимому. Помимо `БЗ` и `бланк-заявка`, документ считается БЗ, если в его тексте есть фраза `СВЕДЕНИЯ ОБ ИСПОЛЬЗОВАНИИ ПРОИЗВЕДЕНИЙ`.

## Yandex Vision OCR

OCR кадров выполняется через Yandex Vision. Сетевые сбои OCR не должны останавливать весь pipeline: для каждого кадра выполняются повторные попытки, а при неуспехе ошибка записывается в `Frame_OCR.txt`, `All_Frames_Text.txt` и `OCR_Log.json`.

Настройки через переменные окружения:

- `YANDEX_VISION_RETRIES`: количество попыток для одного изображения, по умолчанию `3`;
- `YANDEX_VISION_TIMEOUT`: таймаут одного запроса в секундах, по умолчанию `90`.


## OpenAI / ChatGPT

OpenAI/ChatGPT временно отключён по умолчанию, чтобы не тратить токены во время настройки Python-проверок. Основной pipeline пишет заглушки в файлы OpenAI-результатов и не делает сетевые запросы к OpenAI. Для включения задайте:

```text
ENABLE_OPENAI=1
```

## Поиск актёров для id=14

Проверка `id=14` берёт лица из `Faces_Thumbnails.pdf`: каждая страница PDF считается отдельным лицом и проверяется отдельно. Если PDF отсутствует, используется резервный источник `Clipchecker_materials/faces`.

Для каждого лица программа сначала пытается распознать актёра по локальной базе эталонных фото через `face_recognition`. Если совпадение найдено, имя берётся из базы и выводится в интерфейс. Если локального совпадения нет, программа использует самостоятельный прямой веб-поиск без сторонних API: пробует загрузить локальный файл через обычную веб-форму Google reverse image search, а при наличии публичной ссылки дополнительно открывает обычные страницы Google Lens / Google Images / Yandex Images. Из HTML найденных страниц извлекаются возможные имена.

Локальная база медийных лиц по умолчанию ожидается здесь:

```text
C:\Users\PTambulatov\Desktop\PT\2.Расчеты\50. ХАКАТОН\7. face recognition
```

Структура базы плоская: фото лежат прямо в этой папке, имя файла является именем медийной личности.

```text
C:\Users\PTambulatov\Desktop\PT\2.Расчеты\50. ХАКАТОН\7. face recognition\
  Иван Ургант.jpg
  Лариса Гузеева.jpeg
  Константин Хабенский.jpg
```

Путь можно переопределить через `MEDIA_PERSON_KNOWN_FACES_DIR`. Порог совпадения задаётся `MEDIA_PERSON_KNOWN_FACE_TOLERANCE`, по умолчанию `0.5`. Чем ниже значение, тем строже сравнение.

Для Google upload публичная ссылка не обязательна. Для Google Lens / Yandex Images по URL лицо должно быть опубликовано одним из способов:

- `MEDIA_PERSON_PUBLIC_FACE_BASE_URL`: публичная базовая ссылка на папку с файлами лиц; программа добавит к ней имя файла.
- `MEDIA_PERSON_FACE_UPLOAD_ENDPOINT`: внутренний upload-endpoint, который принимает multipart-поле `image` и возвращает JSON с `url`, `image_url`, `public_url` или `link`.
- Yandex Object Storage: нужны `YANDEX_BUCKET` и S3 static keys `YANDEX_S3_ACCESS_KEY_ID` / `YANDEX_S3_SECRET_ACCESS_KEY` или стандартные `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY`. При наличии `boto3` файл будет временно загружен в бакет и передан в поиск по публичной ссылке.

SerpApi больше не включается автоматически только из-за наличия `SERPAPI_API_KEY`. Для явного API-резерва нужно задать `MEDIA_PERSON_ENABLE_SERPAPI=1` или `MEDIA_PERSON_SEARCH_PROVIDER=serpapi_google_lens`. Если публичный URL для лица получить нельзя, пункт `id=14` останется зелёным с сообщением `Актер не распознан`, а причина будет записана в `Actor_Web_Search_Results.json`.
